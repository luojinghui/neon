# 实现聊天页面和功能

## 前置动作

现在请分析src/app/soul 和src/server.js的代码，我需要开始开发soul聊天功能。

## 现状

只生成了聊天列表，点击后提示功能未开发；

## 需求

1、聊天列表页面现在展示所有公开的聊天室数据（使用默认的公共id），后续将增加个人列表数据（通过登录用户id关联的1v1，1v多聊天室），本期不考虑；
2、聊天室支持加密和不加密，如果监测到加密聊天室，需要弹窗输入密码后验证是否可以进入；
3、非加密或者已经验证密码的方式可以跳转到聊天室详情页（/soul/${roomId} 路由是聊天室详情页）；
4、聊天室布局使用现在默认的布局方式，顶部使用通用的topbar，中间展示聊天室名称，右边展示主题切换；
5、聊天室是中间聊天记录列表（左边是远端，右边是本地，显示名称和头像（随机头像）；聊天内容、时间，点击后旁边出现操作功能（类似dropdwon），默认展示复制、下载、更多按钮），底部包含两部分（一行是输入框、发送按钮，占满一行，一行是扩展区域，支持表情、文件、语音通话、视频通话等mini icon，显示方块icon，不需要文字说明），界面整体优雅简洁，符合现代聊天页面的美感。设计颜色参考现有的主题色配置方案；
6、第一期只画对应UI和实现基础的聊天功能（进入聊天室，查找历史消息（加缓存），发送文本消息，其他功能均不实现）
7、每个浏览器环境如果不存在登录信息，则直接创建一个默认账号信息，权限是游客，账号信息中的id不能重复，需要静默查询一次数据库，避免userid重复，登录后重置权限是普通用户，关联用户名；
8、需要在列表和详情页均作权限判断和账号处理，如果遇到非权限访问，则提示并强制跳转到列表页面；
9、所有聊天界面均在加载后和聊天时缓存历史数据，后续进来后优先加载历史数据，再同步更新近50条数据，更多数据才用点击加载更多访问，默认是50条一次；
10、技术要求：server已经实现了socketio，直接使用即可；
11、聊天消息第一期支持文本消息，后续扩展链接、图片、视频、音频、markdown、文件、音乐卡片等
12、所有核心非UI逻辑，请单独在core/index.ts按照class类进行管理（参考cloud的src/app/cloud/core.ts处理逻辑），避免UI中存在大量逻辑；
13、采用zustand来处理状态；
